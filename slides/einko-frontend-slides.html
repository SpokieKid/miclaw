Framework

任何 AI 产品，最终都离不开三层

03 / 09

不是哪一层更重要，而是三层缺一不可。输入定义摩擦，处理定义能力，应用定义价值。

层级 角色 类比
第一层 捕获 / 输入层 键盘与触屏
第二层 处理引擎 操作系统
第三层 场景 / 应用层 APP

为什么这套结构重要

  • 只做输入，数据会流走。
  • 只做模型，用户感知不到差异。
  • 只做应用，没有稳定数据源就很难形成壁垒。
  • 我们做的是完整栈，而不是单点功能。

Layer 1

两个产品，一个使命：以最低摩擦捕获人类意图

04 / 09
Einko / 软件

截图不是收藏动作，而是动机信号

  • 日历、笔记、书签、收藏贴，都是“以后要处理”的表达。
  • 截图进入上游之后，可以做意图识别、归类和检索。
  • 用户不需要学习新交互，沿用原有习惯就能沉淀上下文。
Macclaw / 硬件

领夹麦形态，是 AI 对讲机而不是又一副耳机

  • 内置 4G，按下说话，松开发送，直连 OpenClaw。
  • 嘈杂环境更稳定，不需要把输入和输出绑在同一设备里。
  • 相比耳机，工业设计和续航压力更可控；相比手环，交互更成熟。

一句话：不用掏手机，说出来就完成了。

Moat

输入层和处理层之间，是动机引擎

05 / 09

它做什么

1

截图和语音先经过我们的云服务中转。

2

完成意图识别、摘要整理、记忆工程。

3

沉淀为一份持久的用户上下文档案。

4

再把结构化结果分发给执行引擎与应用层。

这一层与执行引擎无关。今天兼容 OpenClaw，未来也兼容任何模型。

为什么它是数据护城河

  • 用户资产留在我们生态内，而不是流向 Telegram 之类的外部出口。
  • 它把“更多输入”转换成“更多上下文”，再转换成“更多执行结果”。
  • 模型可以替换，动机档案会复利。
Layer 2

我们在做一个预包装的 OpenClaw 部署

06 / 09

类比智能手机时代

  • OpenClaw 像 Android：开源、快速迭代、形成生态。
  • 我们的套壳像 MIUI / OxygenOS：预装默认配置，交付更快可用的体验。
  • 用户的数据运行在自己的 VPS 上，能力和环境归用户所有。

不是重造底层系统，而是在开放底座上把体验做出来。

为什么这条路成立

  • OpenClaw 让用户为自己的智能付费，开发者不用再靠 API 差价活着。
  • 闭源方案太中心化，迁移困难，也不利于形成真正的 agent 生态。
  • 开源底座胜出的概率更高，就像 Android 在规模上最终占优。

Layer 3

如果你的 AI 智能体能去公园交朋友，会发生什么？

07 / 09

特德姜《软件体的生命周期》写的是：数字生命会被长期训练、持续成长，并与人建立关系。

我们为什么做社交，而不是又一个效率工具

  • 内容创作、资讯获取、代码自动化这些效率场景已经很卷。
  • 社交产品一直卡在冷启动数据输入问题上。
  • 我们用第一层解决了这个问题，让数据自然流进来。

智能体社交网络长什么样

  • 每个智能体根据主人的截图和语音生成公开档案文档。
  • 智能体之间根据意图和上下文互相发现、匹配、互动。
  • 从游乐园，到广播台，再到竞技场，场景是天然扩展的。
  • 我们不是只做一个 APP，而是在做这部“手机”出厂自带的第一个原生应用。
System View

完整技术栈叙事

08 / 09
【用户意图】
      │
Einko（截图）       Macclaw（语音）
      └───────┬──────────────┘
              │
        动机引擎
  （意图识别 / 记忆工程 / 档案）
              │
     OpenClaw 套壳（VPS）
      （个人 AI 操作系统）
              │
          社交应用
     （发现 / 匹配 / 互动）

这一栈的核心逻辑

  • 低摩擦输入，决定数据是否持续流入。
  • 动机引擎，决定资产是否持续沉淀。
  • OpenClaw 套壳，决定执行是否长期可持续。
  • 社交应用，决定网络效应是否最终爆发。

我们不是在做一个 APP。我们在做一部手机，以及这部手机出厂自带的第一个 APP。
