设计问答

为什么是领夹麦，不是耳机？
耳机工业设计复杂、续航难、当代年轻人主要用来听歌。领夹麦和手环低头/抬手按下即用，嘈杂环境也能用，BOM 极低。
没有扬声器，怎么看结果？
故意区隔「下达指令」和「验收结果」。结果沉淀在 EINKO APP — 数据资产留在我们的生态，而非流向 Telegram。

战略判断

为什么我们押注开源阵营 — OpenClaw 就是 Android

苹果和安卓的竞争已有定论。在 AI 操作系统的战争里，我们判断开源阵营会胜出 — 我们选择做这个生态里最好的手机厂商。
闭源阵营 · 苹果路线

Anthropic / OpenAI 的野心

  • Claude Code Work — Anthropic 尝试过的闭源 Agentic 平台
  • GPT 应用商店 — 想做 App Store，但 GPT 本身不是 agentic model
  • 体验统一，但生态封闭，开发者受限
  • 开发者面临双平台负担（类比 iOS + Android 两套代码）
  • 平台抽成，Token 定价权在厂商手中
历史证明：苹果的出现是意外，不是必然。在足够大的市场，闭源通吃是不可能的。 安卓最终市场份额远超苹果。
开源阵营 · Android 路线

OpenClaw 的胜出逻辑

  • 完全开源，持续迭代，社区驱动
  • BYOK — Token 消耗责任转让给用户，开发者轻装上阵
  • 任何人都可以在上面做定制 UI（我们就是其一）
  • 多厂商并存，市场份额不会被一家独占
  • 长期来看，开源生态更符合市场规律
我们的位置：OpenClaw 生态里的「小米」— 做最好的定制化 UI 层，同时做「米聊」（Agent 社交网络），保持对所有阵营的中立性。
如果 AI 操作系统的战争从一开始就是开源阵营领跑，那世界上可能根本不需要「苹果路线」— 我们押注这个判断。

数据护城河

动机引擎 — 越用越懂你的数据飞轮

我们不只是一个 AI 工具的入口，更是一个积累用户动机数据的引擎。无论底层 AI 如何变化，动机引擎都让用户越来越离不开我们。

动机引擎是什么

1
用户截图和语音输入 → 经过我们的云服务中转
2
语音转文字 + 内容整理 + 动机识别与标注
3
沉淀为用户「动机档案」 — 存在我们的数据库
4
越用越懂你 → 更准确的意图理解 → 更好的 Agent 匹配
为什么要「中转」而不是直连 OpenClaw？
中转的价值：我们积累的不是 AI 回答，而是用户的「想要什么」— 这才是真正有价值的数据资产。
类比：Google 不拥有网页内容，但拥有「用户想搜索什么」的数据。

平台中立性 — 护城河的关键

动机引擎与执行引擎解耦
第二层换成国产模型？→ 动机数据不丢
OpenClaw 停更了？→ 切换执行引擎，动机档案仍在
竞品复制我们的硬件？→ 复制不了积累多年的动机数据
用户想换平台？→ 数据迁移成本 = 天然锁定

飞轮效应

更好匹配

投资人问答

常见问题解答

创始人对核心疑虑的深度思考

Q1：为什么是领夹麦，不是耳机？

耳机工业设计极复杂（续航、miniaturization、音质），当代年轻人主要用耳机听歌而非交互。领夹麦和手环两种形态都在评估 — 核心是「低头 / 抬手按下即用」，嘈杂环境也能用，BOM 极低。我们不想把硬件做复杂了。

Q2：没有扬声器，用户怎么接收信息？

故意设计。「下达指令」和「验收结果」是两个时间节点，不需要即时反馈。结果会推送到 EINKO APP — 这样数据资产留在我们的生态里，而非流向 Telegram。早期也可发 Telegram，但终态是 EINKO APP。

Q3：你们是否强依赖 OpenClaw？

是，但这是战略选择，不是弱点。OpenClaw 是这个时代的 Android — 开源、开放、BYOK。我们是其上的「小米 UI」，即使 OpenClaw 大改，我们的适配成本可控。更关键的是：动机引擎是平台中立的，数据资产不依赖任何特定执行引擎。

Q4：数据沉淀在哪里？

两处：① EINKO APP — 任务处理结果、截图分析、Agent 交互历史，都存在这里；② 动机引擎云服务 — 语音数据经过我们中转，提炼成用户的「动机档案」，存在我们的数据库。这两块数据资产构成最核心的护城河。
