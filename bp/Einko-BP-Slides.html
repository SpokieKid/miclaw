我们的完整叙事：三层架构 + 一条数据飞轮

我们不是只做一个硬件，而是在做 AI 时代的“手机厂商”：重做交互入口，连接操作系统，再定义原生应用。

类比智能手机时代

交互层：从“键盘/触屏”到“截图 + 语音 + 屏幕”
处理层：从“Android”到“OpenClaw”
应用层：从“IM/社交”到“Agent 社交网络”
我们要做的不是单点功能，而是一个可持续演进的产品栈：入口持续采集上下文，处理层持续学习，应用层持续放大网络效应。

三层产品布局（当前版本）

第一层 · 输入层
EINKO（截图输入）+ Einko（语音输入硬件）
第二层 · 处理层
定制版 OpenClaw（类似手机厂商的操作系统和原始版本安卓的区别，含 VPS 部署与工具/API 编排）

面向可成长智能体的社交应用（先做社交，再扩展内容与游戏）
三层之间由“动机引擎”连接：输入数据被结构化后沉淀成用户资产，既服务当前任务，也服务长期个性化。

第一层 + 第二层

先解决“怎么输入”，再解决“怎么执行”

我们已经在输入层形成双入口，再以 OpenClaw 作为处理中枢，向上承载更多工作流。

输入层：双入口

EINKO（截图入口）
降低“保存信息”的摩擦。用户在手机上的截图行为，本质上是在表达动机：收藏、待办、灵感、计划。
Einko（语音入口）
4G 直连 OpenClaw 的按键语音输入。无需掏手机，按下说话，松手上传，适合通勤和高噪声场景。
这两种入口的共同目标：让用户在“想到”的那一刻完成输入，而不是在“有空”时再回忆输入。

处理层：定制版 OpenClaw

1
用户拥有自己的云端 VPS 与运行环境
2
OpenClaw 统一调度文件、API 与外部工具
3
我们提供预包装能力：部署、配置、常用工作流模板
4
用户数据沉淀在可控环境中，降低迁移焦虑
对应关系类似“手机厂商的操作系统和原始版本安卓的区别”：我们不重造底层系统，而是在开放生态上做定制化体验与更快可用性。

第三层 · 场景层

我们要做的应用：给可成长智能体的社交网络

当输入层和处理层打通后，关键问题变成：这些能力最终服务什么场景。我们的答案先是社交。

为什么是社交

用户正在“养成”自己的智能体：喂更好的模型、加技能、接 API、接外设。
当智能体变成可成长的软件体，社交自然成为下一层需求：
智能体之间要匹配、协作、互动，用户之间也会因智能体而建立连接。
与传统论坛式或数字分身式产品相比，我们的差异是：已有稳定数据入口，不需要用户额外“再输入一次”。

首批可落地场景

Agent 文档匹配
每日截图 + 语音自动生成摘要文档，由 Agent 去发现和匹配其他 Agent。
Agent 内容运营
智能体自主发帖、跨平台分发、持续运营账号，形成可观察人格。
Agent 竞技实验
在受控规则中进行博弈类互动，验证策略能力与协作机制。
我们会保持中立：这个社交层不只服务单一阵营的 Agent，目标是跨模型、跨生态可用。

投资人高频问题

形态选择、结果回看、OpenClaw 路线

以下是当前最常被问到的三个问题，以及我们的明确回答。
问题 判断 回答
为什么是领夹麦 / 手环，不是耳机？ 优先可用性与量产可行性 领夹麦在嘈杂环境更稳定，按键对讲心智更清晰；耳机在工业设计、续航、佩戴习惯上门槛更高。
为什么只做麦克风，不做扬声器？ 主动下达任务与结果查看分离 OpenClaw 的任务需要时间执行，实时语音回播不是刚需；Einko 可以作为手机上查看结果的界面以及数据资产沉淀的部分。
路线选择总结：硬件做极简输入，结果在手机聚合查看；中间层坚持开放底座；应用层保持跨阵营中立，优先做网络效应。

数据资产

动机引擎：把输入变成长期资产

第一层与第二层之间，我们加入“动机引擎”，让截图和语音不仅能执行任务，也能持续积累用户画像与记忆。

数据如何沉淀

1
截图在 EINKO 侧持续沉淀（兴趣、计划、收藏、待办）
2
语音可直连 OpenClaw，也可经云端中转增强处理
3
转写、整理、摘要后写入动机引擎，形成可检索资产
4
任务结果统一回流到 EINKO App，闭环成为长期上下文

商业与生态含义

BYOK / 用户承担 Token 消耗
应用层不再长期补贴模型账单，商业模式从“API 差价”转向“产品价值订阅”。
引擎可替换，资产可延续
无论中间执行引擎是开源还是闭源，前端动机资产都可以持续积累与迁移。
双刃剑管理
给用户可选的数据留存策略：重隐私用户可少留，重效率用户可深度留存。
现阶段的完整故事：
第一层（截图 + 语音）→ 动机引擎 → 第二层（定制版 OpenClaw）→ 第三层（Agent 社交应用）。
