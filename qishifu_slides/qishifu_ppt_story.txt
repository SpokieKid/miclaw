1.

就在我们做硬件的过程中，我们其实发现，Openclaw 特别像特德姜小说里的“软件体”, 就是可以简单理解为, 能自主进化, 学习, 社交, 有陪伴属性的智能体, 而且它很可能会像手机一样,新时代人手一个.



2.
为什么会有这个想法呢 ? 得从我们最开始做的是一个叫 Einko 的截图智能助手。

之前我也在启师傅客厅里分享过，简单来说，它有点像 ColorOS 上的“小布记忆”，就是你能够把你的截图转换成日程、笔记或者收藏夹，然后它能够去自动帮你归类和整理。

(插两张图)

3.我们在做的时候遇到了一些困难。

截图是非常好的数据，包含了非常丰富的用户意图，你其实可以根据这些意图做很多有意思的事情。比如有的时候，如果我截取了用户给我提出的 8 个 bug 或 feature 反馈，按理说它应该有一个自动存到 Linear 上的功能，而不需要我手动去操作。

但在实践过程中，这会面临两个主要挑战：

工作量适配：我们要去支持每一种工作流，都需要做一定的适配工作。
Token 消耗：如果我们把它做成一个非常通用的 Agent，用户的订阅费完全抵挡不了 Token 的消耗量。因为大家都知道，如果你去做一个像 Claude 这样通用的 Agent，是非常消耗 Token 的。

场景价值与付费意愿的问题

我们一直会聊到，很难给咱们的截图找到一些高价值的场景。日历虽然是一个非常好的场景，但它很难直接生产出什么东西来，所以用户的付费意愿不会特别高，它不会是一个刚需。

我们意识到，如果想要去做高价值的场景，你需要一个非常强大的智能体，而且还得是用户愿意为之付费的。

如果我们沿着这个方向走，会发现自己做的东西总会越来越像一个通用 Agent，而这肯定不是我们想要的。因为我们不想把这个产品做得特别重，我们想要的是一个非常轻量的东西。

6. 然后就遇到了 OpenClaw 的爆发。

其实 OpenClaw 的爆发并不是一个意外，因为如果去年大家在用 Claude Code 的话，会意识到 Claude Code 就是一个通用 Agent,  OpenClaw 是一个更激进、权限更丰富, 目标人群更多的 CC。所以 OpenClaw 类的产品几乎是一定会出现的


7. 我们一直在等着那个特别好的产品形态就是 Openclaw。

 OpenClaw 有三个特点？

首先，它自己是一个智能体，然后用户 Token 的消耗由用户自己负担。我觉得这个是对一个应用开发者来说非常好的 feature。

因为以前我在“启师傅客厅”的时候，我会跟启师傅开玩笑说，我们每次做 AI 应用的感觉，都有点像是在给 API 厂商打工的。

具体原因如下：
我们要做非常多的工程化工作来分销 API。
到头来你还不一定能赚到钱。


8.
其次，开发者的心智不再需要变成“我要负责从用户的数据输入、处理，再到找场景的每一步都想得特别好”。这样上下文是分散在不同的 agent 里的.
我们所有人都希望用户尽可能所有的数据都放在一个地方. 这个地方一定不会是一个应用, 在现在这个地方就是手机和电脑这样的端侧设备.



工作流，或者某个特定领域的垂直 Agent 还是会持续存在。比如像 Lovart，或者一些专门针对特定场景的工作流，它们依然有生命力，因为它们在特定场景下做得非常特化非常好。

但我觉得未来确实会逐渐趋近于：我们会更愿意把所有的数据存放在一个地方。

这个地方通常是一个比较受保护的（可能会像你的手机一样），它有很多的隐私保护和策略。如果应用要去使用这些数据的话，你不会说让每个应用都能访问到手机上的全部数据，而是会采用一种更温和、或者说更协商的方式来使用这些数据。

9.

所以这个时候我们就想清楚了，我们要做的事情其实就两件事情：

提供更简便、摩擦阻力更低的交互方式, 这个 Arvin 已经讲过了


10. 

第二件我们要做的事情就是去帮用户的这些数据找场景，也就是开发 App。 这里的数据不仅仅是通过 麦克风或者截图获得的数据, 也包含了用户在使用 openclaw 的时候的其他输入, 就和现在开发手机应用是一样的.


我们默认未来大家自己的这些软件体上会有很多各种各样的数据。这些数据可能是从 Einko 或者 Einko Mac 里面来的，也可能是从你自己其他地方（比如 Gmail 或者 whatever 你其他地方）收集拿到的数据，都会放在这里。

我们可能只会用其中一小部分，去做一些譬如说自动化的工作，也可能是和社交相关的事情。

11. 

至于中间那一层，即软件体到底要怎么交到用户手里。如果没有 OpenClaw 怎么办？我觉得这是一个顺便做的事情, 应该是卖硬件还是卖云端 vps, 这个我暂时还没法知道哪种会胜出, 我可能会倾向于云端一点.

像现在的 Kimi、火山，还有很多国外的云服务商（SaaS），大家都会提供自己云部署版本的 OpenClaw。我们可能不会那样做，我们可能只会做一些非常简单的功能。

我们自己应该也会做一个非常简单的云端部署的服务, 用来配合我们的输入交互方式, 以及未来想要设计的应用.



12. 我们特别想做的是社交类的 app, 而社交 app 能说明为什么 openclaw 特别好.

用过很多 AI 社交产品，我觉得很大的问题是用户没有办法去给它提供上下文。我必须要打开一个 APP，然后去填写个人信息，它才能够根据我的信息画像去做社交。

但问题在于：
门槛非常高
用户为什么要为了一个 APP，就把自己珍贵的、关于个人性格的东西交给它？
无法通用
如果我要使用三个不同的 AI 社交产品，我就需要在三个不同的 APP 里面录入三遍。

这其实是一件非常耗时耗力的事情。我觉得这是目前 AI 社交产品最大的问题：开发者没有办法拿到足够多的用户 context，因为用户不会只为了你这一个社交应用去提供 context。

只有工作 生活兼备的场景才能让用户自然地在其中积累数据, openclaw 就恰好满足这个条件. 既能帮你打工, 还能帮你找朋友.


13.
我觉得 Moltbook 是一个特别好的案例。为什么说 Moltbook 是艺术品？虽然它可能有一些虚构的成分在里面，但它有两个核心优势：

内容生成成本的转移
Modebook 上面的内容生成完全不消耗开发者的钱。你像现在我们看到的一些 AI 产品，它们消耗的 Token 量其实是由开发者来承担的，而不是由用户承担，这其实是一件非常“烧钱”的事情。但在 Modebook 上，你写篇帖子是完全靠用户自己的想法去创作，而不是公司去生成的，这件事情很重要。

对本地数据的访问能力
Modebook 是直接根据你电脑上的数据来运行的。你有能力访问到电脑上已经存在的大量数据，而不是让用户为了用你的应用，专门去录入或者下载数据，情况不是那样的。

所以这里有一个好处：当我们把视线放得更长远一点，如果我们的一个应用不需要端到端地负责“从获取上下文到处理、再到应用场景”的全过程，其实思路就会打得非常开。

14.我认为 OpenClaw（包括现在和未来）还一定要做的一件事情，就是对用户数据的权限管理。

比如什么样的 Skills 能够访问哪些数据，以及是否会有一些个人数据的交换协议，这些都是未来特别值得去做的事情。我最近有看到一些 issue 在讨论类似的话题。

我觉得它特别像一个新时代的手机，我们要做的就是给相应的个人数据找场景。当我们使用这些数据时，不应该只想到单一的场景或单一的数据源；它包含的可能是非常多自然生成的、或者主动输入的数据，这些数据都会汇聚在一起，形成一个非常庞大的上下文库。

15. 找场景要怎么做呢？我们会把这个东西叫做一个“动机引擎”。

我们手头其实有非常多用户的“潜意识数据”——比如截图，或者平时记录的一些零碎想法。这些数据往往是不成型的、琐碎的，用户可能不知道该怎么把这些散乱的想法组装成一个成品。

“动机引擎”的核心概念，就是如何从这些零碎的想法中，匹配到对应的工作流, 而且这一步一定要争得用户的同意, 否则用户一开始就不会愿意给你。比如：
我左边有一些特别零碎的数据，像每天记录的选题库，或者随手收集的优秀产品文章。
动机引擎能自动从中进行匹配，连接到成熟的工作流（如小红书发布工作流或 Newsletter 写作工作流）。

通过这种方式，让 AI 来告诉用户：每天记录的这些碎片内容，究竟可以拿来生成什么样的产品或内容。这种浅层、小量的“动机”，其实就蕴藏在我们的截图或语音里面，而我们要做的就是把它们高效地匹配起来。

16.所以最终总结我们的事情就很简单，就是我们想要让大家更可能地捕捉到个人生活中的信息，可能是潜藏的灵感、意识或其他类似的东西。

我们要寻找特别好的方式去使用个人数据，将其转化为各种各样的创造，或者完成自动化、提升效率的场景。
